# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

Auto-generated Rust client for the [Stedi APIs](https://www.stedi.com/docs). Stedi ships one
OpenAPI spec per API; each is vendored into its own module under `src/<service>/` (generated by
openapi-generator). Only `src/lib.rs` and `src/client.rs` are hand-written.

## Architecture

- Each Stedi API → one module (`claims`, `core`, `enrollment`, `event_destinations`, `healthcare`,
  `manager`, `payers`) containing its own generated `apis/` and `models/`. Keeping them separate
  avoids model-name collisions across specs and lets each be a Cargo feature.
- The rust generator emits absolute `crate::apis` / `crate::models` paths. `generate.sh` rewrites
  these to `crate::<service>::…` after copying the generated code into `src/<service>/`, so it
  compiles inside a submodule. **This rewrite is the one subtle part — preserve its ordering.**
- `src/client.rs` holds a `StediClient` that stores the API key and, via a macro, exposes a
  per-service accessor returning that service's `Configuration` (base URL from the spec + API key
  in the `Authorization: Key <key>` header).

## Commands

```bash
cargo check --all-features                                  # build / type check
cargo test --all-features                                   # run tests
cargo check --no-default-features --features "claims,native-tls"  # validate feature gates
./generate.sh                                               # regenerate from latest specs
cargo doc --no-deps                                         # generate docs
```

## Regeneration

`./generate.sh` fetches all seven specs from https://github.com/Stedi/openapi, records their
combined SHA-256 in `SPEC_HASH`, generates each into a temp dir, vendors `apis/` + `models/` into
`src/<service>/`, rewrites crate paths, writes each `src/<service>/mod.rs`, and runs `cargo check`.
The run is idempotent. Do not hand-edit anything under `src/<service>/` — fix the spec upstream or
the `generate.sh` post-processing instead.

The generator version is **pinned** via `GENERATOR_VERSION` in `generate.sh`; the JAR is fetched from
Maven Central (cached under `~/.cache/openapi-generator`), so local and CI produce identical output
with only a JDK installed — no brew/npm generator needed. Bumping `GENERATOR_VERSION` is deliberate:
it can change generated output (e.g. the String→`chrono` date-time switch in 7.15+, which is why
`chrono` is a dependency). Regenerate and review the diff when changing it.

CI: `update-spec.yml` runs `generate.sh` daily; on any change it bumps the patch version and opens a
PR. Merging a `update/stedi-spec-*` PR triggers `tag-release.yml`, which tags the release and then
publishes to crates.io (`cargo publish`, gated on the tag being newly created). Publishing needs a
`CARGO_REGISTRY_TOKEN` secret. Note: publish runs as a second job in the same workflow because a tag
pushed with the default `GITHUB_TOKEN` would not trigger a separate `on: push: tags` workflow.
